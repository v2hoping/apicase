# 产品概念模型

> 这是 apicase 的奠基性产品设计，全局有效，所有需求据此展开。

## 产品设计原则

以下原则为 apicase 的**第一性约束**，任何需求、设计与实现都不得违背；冲突时以本节为准。

1. **工作空间即文件目录，一切皆文件（local-first）**
   - 工作空间就是一个本地文件目录，所有数据以本地文件系统为唯一载体，**一切皆文件**。
   - 天然支持 **Git 同步 / 版本管理**，可像普通软件项目一样在**任意位置**打开、迁移、分享。

2. **零登录，开箱即用**
   - **不引入任何登录 / 账号 / 联网授权**环节；安装即用，离线可用。

3. **极致性能**
   - 作为本地桌面软件，必须确保**操作流畅、内存占用小、响应快**；性能是基础体验，不是可选项。

4. **界面现代化**
   - 界面**美观、简洁、现代**；视觉与交互参考 **Postman**。

5. **YAML 描述 case，贴合行业惯例，数据归用户私有**
   - 用 **YAML** 描述 case；其中 api（request）与 flow 的格式**尽量符合行业惯例**，降低用户学习成本。
   - **数据永远归用户私有**——不上传、不托管，完全留在用户本地。

## 核心抽象：case = DAG

把「用例（case）」建模为 **DAG（有向无环图）**——节点是单个 API 请求，边表示执行顺序与数据依赖。

- **单个 API 调试 = 只有一个节点的 DAG**。
- 单 API 与编排用例在模型上**统一**，从单请求扩展到多节点编排平滑、无断层。
- UI 自适应，落成**三视图切换（文本 / 流程 / 请求）**：
  - **文本** = 原始 YAML，**互斥模式**（不与结构视图并存，规避「两个真相源」同步问题）；也用于校验不通过时的兜底。
  - **流程** = DAG 画布（`dependsOn` 自动分层布局，支持 fan-in），**请求** = 复用的 Postman 式请求编辑器，二者可并存。
  - **内容驱动默认视图**（仅打开时决定）：非法/非 case → 文本；单请求 → 请求；多请求 → 流程 + 请求。
  - 内部统一 steps 模型（单节点 = 1 步），单 → 多提升平滑；见 [1.feature/20260630-flow画布与三视图](../1.feature/20260630-flow画布与三视图/技术方案.md)。

## 文件即数据（local-first）

打开一个目录 = 一个 apicase **工作空间（workspace）**，一切以文件系统为准：

| Postman 概念 | apicase | 说明 |
|---|---|---|
| Workspace | **工作空间（打开的目录）** | 打开一个目录即一个工作空间，是根容器 |
| Collection | **folder（目录）** | 目录即分组，天然层级 |
| Request | **case（单节点）** | 默认即只有一个节点的 flow |
| Flow | **case（多节点）** | 需要时把 case 扩展为多节点 DAG |
| Environment | **`application.yml` 中的配置项** | 见下文 |
| Spec | （后续）OpenAPI 契约 | 暂不做 |

**对用户暴露的概念极少**：打开的根目录即**工作空间（workspace）**，其内只有 `folder`（分组）与 `case`（用例）两类，心智负担最小。

参考标杆：**Bruno**（file-based、Git 友好，理念同源）；**Postman**（UI / 功能 / 布局）。

## 关键设计决策

1. **不支持拖拽排序**。与文件系统一致；需要排序时用命名数字前缀（如 `0.`、`1.`）。理由：统一、最低理解成本。
2. **不引入 folder 元信息**。保持目录纯粹，用户注意力集中在文件本身。
3. **case 用通用 `.yml` 后缀**，不用专属后缀。
   - 无需读内容即可组织：检索 = 普通文件检索，**只有点击打开的那一刻才解析**。
   - 降低认知：直接是标准 YAML，「没有独家语法，不用怕」。
4. **文件识别规则**：工作空间根 `application.yml` = 工作空间配置文件；其余 `.yml` 在点击时按 case 解析、渲染，否则按普通文本 / 给出提示。
5. **environment（环境变量）**：
   - 工作空间根 `application.yml` 承载工作空间级配置，`environment` 是其中一个配置项。
   - 支持多套环境、可切换（dev / test / prod…）。
   - **工作空间级配置优先于系统级配置。**
   - 变量引用沿用 `{{var}}` 语法，便于从 Postman 迁移。
6. **存储格式：YAML**。人可读可写、支持注释、Git diff 友好，与 OpenAPI / CI 生态一致。
7. **SPEC（OpenAPI 契约）= 后续能力**：导入 OpenAPI 生成 case，或由 case 反向生成 / 校验契约。
8. **根容器命名 = 工作空间（workspace），不叫「项目」**：
   - 取 **VSCode** 的语义——workspace 指「打开的那个目录」本身，纯 local，**不含云协作含义**（区别于 Postman 那种协作型 workspace）。
   - VSCode 官方明确不使用 "project" 概念（*"VS Code doesn't have the concept of a 'project'…"*）；apicase 形态（打开目录、`application.yml` ↔ `.vscode/`、单 / 多目录）与之同源，故对齐 workspace。
   - 英文统一用 `workspace`，中文统一用「工作空间」。
   - 后续若引入"多工作空间并存 / 协作云"，workspace 仍是自洽的根概念，无需返工。

## case 文件结构

request 结构在单/多节点**完全复用**：单节点用顶层裸 `request:`（语法糖，归一化为单 step），多节点用 `steps:` + `dependsOn`（DAG 依赖指针，不写显式 edges）。**解析判别**：有 `steps` → flow；否则有 `request` → 单节点。变量统一 `{{var}}`。

字段命名参考业界标准（降低学习成本，原则 5）：request 借 **Postman v2.1 + HAR + Insomnia v5 + Bruno**；flow 借 **Arazzo 1.1**（OpenAPI 官方 workflow 标准）+ **GitHub Actions** `needs`。详见 [1.feature/20260630-case读写与格式](../1.feature/20260630-case读写与格式/技术方案.md)。

```yaml
# 单节点 case（已实现读写）
apicase: "0.1"
name: 查询用户                  # 可选，缺省用文件名
request:
  method: GET
  url: "{{baseUrl}}/users/1"   # 单字符串、不含 query
  query:                       # 数组 + enabled（默认 true 省略）；支持重复键/禁用/顺序
    - { name: page, value: "1" }
    - { name: draft, value: "true", enabled: false }
  headers:
    - { name: Accept, value: application/json }
  auth: { type: bearer, bearer: { token: "{{token}}" } }   # type 判别 + 分桶
  body: { type: json, json: { name: Ada } }                # type 判别 + 同名值子键

# 多节点 flow（已实现可视化编辑 + 执行：变量透传 / 输出提取 / 断言）
steps:
  - id: login
    request: { method: POST, url: "{{baseUrl}}/login" }
    outputs: { token: $.data.token }      # JSONPath 提取
  - id: createOrder
    dependsOn: [login]                    # DAG 依赖
    request:
      headers:
        - { name: Authorization, value: "Bearer {{ steps.login.outputs.token }}" }
```

- **body 多类型**：`type` ∈ `none|json|text|form-urlencoded|form-data`，值放同名子键（`json` 存结构化 YAML，diff 友好）。
- **序列化裁剪**：`enabled:true`、`auth:{type:none}`、空 body 等默认值不落盘，UI 生成的 YAML 简洁、diff 噪声最小。
- 工程上：本阶段前端用 `js-yaml` 解析（`case.ts`）；后端 `serde` 统一模型推迟到 flow 执行引擎阶段。与 Postman 互操作时走 JSON。
