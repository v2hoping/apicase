# 技术架构

> 描述 apicase 的实现架构现状（v0.1）。

## 技术栈

- **桌面框架**：Tauri 2.x（Rust 后端 + 系统 WebView 前端）。
- **后端**：Rust；`reqwest`（rustls TLS，避免本地 OpenSSL）；`serde` / `serde_json`；`tokio`（仅测试）。
- **前端**：React 19 + TypeScript + Vite 7。
- **前后端通信**：Tauri command + `invoke`。

## 后端：核心命令与数据模型

数据模型（`serde`，对前端以 camelCase 序列化）：

```rust
struct HeaderEntry { key: String, value: String }
struct ApiRequest  { method, url, headers: Vec<HeaderEntry>, body: Option<String> }
struct ApiResponse { status, status_text, headers, body, elapsed_ms }
```

- **`send_request(request: ApiRequest) -> Result<ApiResponse, String>`**：Tauri 命令。
- **`perform_request`**：实际发请求逻辑，与 Tauri 解耦，便于单元 / 集成测试。
  - 由 Rust 后端用 `reqwest` 发出，**天然绕过浏览器 CORS**。
  - 校验空 URL、非法 HTTP 方法；统计耗时（请求发出到响应体读完）。

## 前端：交互

请求表单 → `invoke("send_request", { request })` → 后端发请求 → 渲染响应。
UI 对标 Postman：方法下拉（彩色）+ URL + Send；Headers（自动增行）/ Body 配置 Tabs；响应区状态徽章 + 耗时 + 大小，Body(Pretty/Raw) / Headers Tabs；左侧 Collections 文件树占位。

## 运行 / 构建 / 测试

```bash
cd app
npm install               # 首次
npm run tauri dev         # 启动桌面应用（热重载）
npm run tauri build       # 打包安装包

npm run build             # 前端类型检查 + 打包
cargo test --manifest-path src-tauri/Cargo.toml   # 后端测试（real_get 需联网）
```

## 当前实现边界

- 仅「单请求」执行，**无文件持久化**（打开目录 / 读写 `.yml` case 为下一需求）。
- 响应按文本处理；大 / 二进制响应、流式留后续。
- YAML 库待引入时采用维护中的 fork（`serde_norway` / `yaml-rust2`），规避 `serde_yaml` 已停维护与 YAML 隐式类型坑。
