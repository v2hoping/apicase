# case 读写与格式（参考业界标准）

> 需求时间：2026-06-30。在「工作空间与界面外壳」之上，打通**文件树 ↔ 请求构建器**：点击 `.yml` case → 解析渲染 → 编辑 → 写回；并把 case 的 YAML 格式参考业界标准**一次定稿**。

## 背景与问题

当前文件树能浏览、请求构建器能发单请求，但**点击 case 只高亮、不渲染**——两块从未打通，local-first「文件即数据」闭环缺失最后一环。本需求补齐该闭环，同时确立 case 的 YAML schema，避免后期迁移。

## 需求内容

### A. case 读 → 渲染
- 点击文件树中 `.yml`（非工作空间根 `application.yml`）→ 读取并解析 → 填充请求构建器（method/url/query/headers/auth/body）。
- 多节点（含 `steps:`）的 case 本次**仅识别 + 占位提示「flow 编辑即将支持」**，不报错、不破坏文件。

### B. 编辑 → 写回
- 任一字段编辑标记未保存（dirty，顶部圆点）。
- `Cmd/Ctrl+S` 或 Save 按钮 → 序列化写回原文件 → 清 dirty。

### C. 文件管理（右键菜单）
- 文件树右键：新建 case / 新建 folder / 重命名 / 删除（删除等破坏性操作弹确认）。
- 操作后重新加载父目录刷新树。

### D. 请求构建器对齐 Postman
- 请求配置 Tab 从 Headers/Body 扩展为 **Params(query) / Headers / Auth / Body**。
- Params 与 URL 的 query 双向同步；Body 支持类型选择（none/json/text/form-urlencoded/form-data）；Auth 支持 none/bearer/basic/apikey。

## Case YAML Schema（定稿，事实来源）

设计原则：**request 结构在单/多节点完全复用**；单节点用顶层裸 `request:`（语法糖，归一化为单 step），多节点用 `steps:`。变量统一 `{{var}}`。

**解析判别：** 有 `steps` → flow；否则有 `request` → 单节点；工作空间根 `application.yml` → 工作空间配置。

```yaml
# ── 单节点 case（本次实现读写）──────────────────
apicase: "0.1"                 # 格式版本（借 Arazzo 顶层 arazzo:）
name: 查询用户                  # 可选，缺省用文件名
vars:                          # 可选，case 级变量
  baseUrl: https://api.demo.com
request:
  method: GET
  url: "{{baseUrl}}/users/1"   # 单字符串、不含 query（不学 Postman raw+host[]+path[] 双份真相）
  query:                       # 数组+enabled；enabled 默认 true 时省略
    - { name: page,  value: "1" }
    - { name: draft, value: "true", enabled: false }
  headers:
    - { name: Accept, value: application/json }
  auth:                        # type 判别 + 按类型分桶
    type: bearer               # none | bearer | basic | apikey
    bearer: { token: "{{token}}" }
  body:                        # type 判别 + 同名值子键
    type: json                 # none | json | text | form-urlencoded | form-data
    json: { name: Ada }        # type=json 时存结构化 YAML（diff 友好）

# ── 多节点 flow（本次仅识别+提示，schema 预留）──
steps:                         # 借 Arazzo steps / GHA jobs
  - id: login
    request: { method: POST, url: "{{baseUrl}}/login", body: { type: json, json: { user: admin } } }
    outputs: { token: $.data.token }     # JSONPath 提取（借 Arazzo outputs + Step CI captures）
  - id: createOrder
    dependsOn: [login]                   # DAG 依赖指针（借 Arazzo 1.1 dependsOn / GHA needs）
    request:
      method: POST
      url: "{{baseUrl}}/orders"
      headers:
        - { name: Authorization, value: "Bearer {{ steps.login.outputs.token }}" }  # 跨步引用
ui:                            # 画布坐标与语义分离（规避 n8n 把坐标塞进节点的 diff 噪声）
  nodes: { login: { x: 0, y: 0 }, createOrder: { x: 220, y: 0 } }
```

**序列化裁剪**（借 Insomnia v5「省略空值/false/默认值」）：`enabled:true`、`auth:{type:none}`、空 body / 空数组等不落盘，UI 生成的 YAML 简洁、diff 噪声最小。

### 字段来源（为什么这样定）

| 部分 | 设计 | 借鉴 |
|---|---|---|
| `url` | 单字符串、不含 query | Bruno / Insomnia v5 / .http（避开 Postman 双份真相） |
| `query` / `headers` | 数组 `{name,value,enabled}` | HAR `queryString[]` + Postman `disabled` + Hoppscotch `active`（支持重复键/可禁用/顺序稳定） |
| `auth` | `type` 判别 + 分桶 | Postman `auth.type` / Insomnia `authentication.type` |
| `body` | `type` 判别 + 同名值子键 | Postman `mode` + Bruno `body:<type>` + HAR `text`/`params` 互斥 |
| `steps` + `dependsOn` | 依赖指针表 DAG，不写显式 edges | Arazzo 1.1 `dependsOn` + GitHub Actions `needs`（diff 小、手写直观、单/多统一） |
| step 内嵌 `request` | 不引用 OpenAPI，自包含 | Step CI（local-first、spec-optional，复用单节点结构） |
| `outputs` + `{{steps.x.outputs.y}}` | JSONPath 提取 + `{{}}` 引用 | Arazzo `outputs` + Step CI `captures` + GHA `${{steps...}}` 的收敛形 |

> 与 Arazzo 的分歧：Arazzo 绑定外部 OpenAPI `operationId`、用 `$`运行时表达式 + JSON Pointer；apicase 改为**内嵌请求** + `{{}}` + JSONPath，更贴 Postman 受众、学习成本更低。字段名对齐 Arazzo，未来可做 apicase ⇄ Arazzo 互转。

## 关键决策

1. **YAML 解析放前端（js-yaml），后端只做通用文件读写。** 本阶段 case 仅「渲染到 UI + 发单请求」，前端解析免 round-trip、迭代快、schema 单点在 TS（`src/case.ts`）。后端 `serde` 统一模型推迟到 flow **执行引擎**阶段（届时调度 DAG 才真正需要后端理解 case 结构）。
2. **单/多节点归一化在解析层完成**；UI 本次只渲染单节点，读到 flow 不破坏文件。
3. 新增前端依赖 `js-yaml` + `@types/js-yaml`；`Cargo.toml` 无需新增依赖。
