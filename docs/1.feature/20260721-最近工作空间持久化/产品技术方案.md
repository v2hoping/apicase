# 最近打开的工作空间持久化

## 需求内容

- 顶栏「选择工作空间」下拉里的**最近打开的工作空间**列表，目前只存在内存（`useState`），
  应用重启即丢。改为**持久化**，重启后仍能看到并快速切回。
- 存储位置采用 **Tauri 应用专用目录**（配置 / 持久数据目录），而非 localStorage：
  - 只按应用 `identifier`（`com.apicase.app`）定位，**与启动方式无关**（dev / 打包 / 浏览器
    不再各存各的、互不相通——localStorage 按 origin 分桶会导致「dev 设的、打包后不见」）。
  - macOS：`~/Library/Application Support/com.apicase.app/settings.json`
  - Windows：`%APPDATA%\com.apicase.app\settings.json`
  - Linux：`~/.config/com.apicase.app/settings.json`

## 执行计划

### 后端（`src-tauri/src/lib.rs`）

引入 `tauri::Manager`（用 `app.path().app_config_dir()`），新增一对**通用**设置读写命令
（读写整份 `settings.json` 原始文本，结构交给前端；后续主题 / 快捷键等设置可复用同一文件）：

- `read_app_settings(app) -> Result<String, String>`：读 `settings.json`；文件不存在返回空串
  （由前端兜底为默认值），其余 IO 错误照常返回 Err。
- `write_app_settings(app, content) -> Result<(), String>`：先 `create_dir_all` 保证配置目录存在，
  再整体覆盖写入。

两命令注册进 `generate_handler!`。文件读写走 `std::fs`（自定义命令内不受 Tauri fs 权限系统限制，
无需改 capabilities）。

### 前端（`src/settings.ts` 新增 + `src/App.tsx` 接线）

- `settings.ts`：定义 `AppSettings { recentWorkspaces: string[] }` 与 `loadAppSettings()` /
  `saveAppSettings()`，封装 `invoke("read_app_settings")` / `invoke("write_app_settings")`
  与 JSON 序列化、字段校验、失败兜底（持久化失败不打断主流程）；另有
  `filterExistingPaths()` 复用 `path_exists` 命令**剔除已删除 / 移动的失效目录**（仅明确
  不存在才剔除，校验抛错保守保留，避免 IO 抖动误删）。
- `App.tsx`：
  - 挂载时 `loadAppSettings()` 读取 →`filterExistingPaths()` **剔除失效目录** → **merge**
    进内存 `recentWorkspaces`（用户在读取完成前可能已打开工作空间，用 merge + 去重避免覆盖丢失）。
  - `recentWorkspaces` 变化时 `saveAppSettings()` 写回；用 `settingsLoaded` ref 门控，
    避免加载完成前用初始空值覆盖磁盘。
  - **失效项落盘清除**：过滤后的列表经写回 effect **整份覆盖** `settings.json`，失效项从文件
    一并删除（非仅显示层过滤）；叠加 `.slice(0, 10)` 上限，双重保证文件不无限累积。
  - **点击时失效兜底（TOCTOU）**：加载时校验通过、但**点击打开这一刻**目录已被外部删除 / 移动的，
    `selectWorkspace` 用 `pathExists` 先校验——失效则从最近列表移除（落盘清除）+ 提示，
    **不切过去、不把失效项重新写回置顶**（修复原 `applyWorkspace` 无条件切换并续命失效项的问题）。
  - **手动删除记录**：下拉里每条最近项 hover 时右侧出现删除按钮（`.ws-recent-del`，opacity 常驻占位
    避免挤动路径文字），`removeRecentWorkspace` 从列表移除——**仅删历史记录、不删磁盘工作空间目录**，
    经写回 effect 同步落盘。删除按钮 `stopPropagation` 避免误触发打开。

### 设置页导航按存储归属分组（配套 UI）

配置页左导航按数据的**存储归属**分两组，中间加分割线（`.settings-nav-divider`），
让「跟随项目」与「跟随应用存储」的设置分开显示——正是本轮存储分层在 UI 上的收尾：

- **跟随项目**（工作空间 / `application.yml`）：通用（工作空间 + 配置文件路径）、环境
- ── 分割线 ──
- **跟随应用**（localStorage / 应用配置目录）：主题、快捷键、关于（「主题」由第 2 位下移到应用组首位）

实现：`SettingsPage` 的 `NAV` 拆为 `NAV_PROJECT` / `NAV_APP`（`NAV` 为二者拼接，供 `section` 类型与遍历）；
渲染时 `NAV.map` 每项返回 `[divider?, button]` 数组（React 自动展平），在 `i === NAV_PROJECT.length`
处注入分割线。
