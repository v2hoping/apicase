# 产品技术方案：脚手架与单 API 调试 MVP

> 本需求为 apicase 的第一个需求。**状态：已落地（2026-06-29），要点已合并进 `docs/0.latest/`。**
> 本文档含两部分：**产品概念模型与设计决策**（全局、奠基性）与 **本期 MVP 需求 + 执行计划**。

## 一、产品概念模型与设计决策

### 1.1 定位与核心抽象

apicase 是一款用于 **API 接口调试、管理与用例编排** 的桌面软件。核心抽象：把「用例（case）」建模为 **DAG（有向无环图）**——节点是单个 API 请求，边表示执行顺序与数据依赖。**单个 API 调试 = 只有一个节点的 DAG**，单 API 与编排用例在模型上统一，扩展平滑、无模型断层。

### 1.2 核心理念：文件即数据（local-first）

打开一个目录 = 一个 apicase 项目，一切以文件系统为准：

| Postman 概念 | apicase | 说明 |
|---|---|---|
| Collection | **folder（目录）** | 目录即分组，天然层级 |
| Request | **case（单节点）** | 默认就是只有一个节点的 flow |
| Flow | **case（多节点）** | 需要时把 case 扩展为多节点 DAG |
| Environment | **`application.yml` 中的配置项** | 见 1.3 第 6 条 |
| Spec | （后续）OpenAPI 契约 | MVP 不做 |

**对用户只暴露两个概念：`folder` 和 `case`。** 心智负担最小，单 API 与编排用例统一为 case。

参考标杆：**Bruno**（file-based、Git 友好的开源 API 客户端，理念高度同源）；**Postman**（UI / 功能 / 布局参考）。

### 1.3 关键设计决策（已确认）

1. **不支持拖拽排序**。与文件系统行为一致；需要排序时用命名数字前缀（如 `0.`、`1.`）自行控制。理由：统一、最低理解成本。
2. **不引入 folder 元信息**。保持目录的纯粹与统一特性，让用户把注意力集中在文件本身，而非额外元数据。
3. **case 文件用通用 `.yml` 后缀**，不使用专属后缀。理由：
   - **无需读内容即可组织**：文件检索 = 普通文件检索，**只有在点击打开的那一刻才解析**，不必预读所有文件判定类型。
   - **降低认知**：直接用通用 YAML，等于告诉用户「不用怕，没有独家语法，就是标准 yaml」。
4. **文件识别规则**：
   - 项目根的 `application.yml` = **项目配置文件**（保留文件名）。
   - 其余 `.yml` 在点击打开时按 **case** 解析、渲染；符合 case 结构即展示为用例，否则按普通文本 / 给出提示。
5. **存储格式：YAML**（确认）。见 1.4。
6. **environment（环境变量）**：
   - 项目根 `application.yml` 承载**项目级配置**，`environment` 是其中一个配置项。
   - 支持**多套环境、可切换**（dev / test / prod…），也可为该项目设置不同配置。
   - **项目级配置优先于系统级配置。**
   - 变量引用沿用 `{{var}}` 语法，便于从 Postman 迁移。
7. **SPEC（OpenAPI 契约）= 后续能力**：导入 OpenAPI 生成 case，或由 case 反向生成 / 校验契约。MVP 不做。

### 1.4 case 文件结构与存储格式（YAML）

**统一 DAG 模型**：单个 case 文件内部统一为 `nodes + edges`。
- 单节点 case = 1 node / 0 edge → UI 呈现「请求视图」（普通请求-响应面板）。
- 多节点 case → UI 切换「画布 / flow 视图」。

**为什么用 YAML**
- 人可读 / 可写、**支持注释**（可给请求字段、用例步骤写说明）、Git diff 友好，与 OpenAPI / CI / k8s 生态一致。
- 业界惯例：数据交换以 JSON 最通用，但**人写的声明式配置以 YAML 为主流**——case / 配置正属于后者。

**工程实现要点**
- 内存用 `serde` 定义统一数据模型；磁盘序列化走 YAML；与 Postman 互操作（导入 / 导出）时再走 JSON——换格式只是换 serde 后端。
- **Rust 生态现状**：`serde_yaml` 官方已归档停维护，需采用维护中的替代（`serde_norway` / `yaml-rust2` 等）；JSON 侧 `serde_json` 为活跃的黄金标准。
- 规避 YAML 隐式类型坑（如 `no`→`false`、版本号被当数字）：schema 校验 + 必要时显式引号。

## 二、本期 MVP 需求

### 2.1 范围

**脚手架 + 单 API 调试最小闭环**。核心是验证「单节点 DAG 执行」这条端到端链路：界面发起一个 HTTP 请求 → Rust 后端发出 → 展示响应。

**脚手架**
- 在 `app/` 下初始化 Tauri 2 项目（前端 + `src-tauri` Rust 后端）。
- 跑通 `tauri dev`，能打开 apicase 应用窗口。
- 约定项目结构与启动 / 构建脚本。

**最小闭环（单 API 调试）**
- 请求编辑：HTTP 方法（GET/POST/PUT/DELETE…）、URL、可选 Headers、可选 Body。
- 发送：前端通过 Tauri `invoke` 调用 Rust 命令 `send_request`。
- 执行：Rust 用 `reqwest` 发起真实请求（由后端发出，天然绕过浏览器 CORS）。
- 响应展示：状态码、耗时、响应头、响应体（JSON 美化 / 原文）。
- 错误处理：非法 URL、网络失败、超时给出可读提示，不崩溃。

**不在本期范围**
- **下一需求优先**：打开目录作为项目、左侧文件树、从 `.yml` 读写单个 case（文件持久化）。
- 之后：多节点 DAG 编排画布、environment 多套配置切换、历史记录、鉴权管理、导入导出、OpenAPI(SPEC)。

### 2.2 界面与交互参考（Postman）

整体 UI 的样式、风格、布局与功能组织 **主要参考 Postman**：
- **整体布局**（目标形态，逐步补齐）：顶部工具栏 + 多标签页工作区；左侧 **文件树**（folder = Collection）；中部请求构造区；下部 / 右侧响应区。
- **请求构造区**：请求行（方法下拉按 GET/POST/PUT/DELETE 彩色区分 + URL 输入 + `Send`）+ 配置 Tabs（Params / Headers / Body，Authorization、Pre-request、Tests 后续补）。
- **响应区**：概要（状态码 / 耗时 / 大小）+ 响应 Tabs（Body Pretty/Raw、Headers，Cookies、Test Results 后续补）。
- **风格**：简洁工具栏、紧凑高信息密度；主色可参考 Postman 橙（`#FF6C37`），但使用 apicase 自有标识。

> 本期 MVP 优先实现 **请求行 + Headers/Body + 响应展示** 主面板；左侧文件树与多标签页可先简化 / 占位，后续需求逐步补齐到完整 Postman 式布局。

### 2.3 验收标准

- [ ] 在 `app/` 下执行启动命令可打开 apicase 桌面窗口。
- [ ] 输入一个真实可访问的 URL（如 `https://httpbin.org/get`）点击发送。
- [ ] 请求由 **Rust 后端**发出（非前端 `fetch`），能成功拿到响应。
- [ ] 界面正确显示状态码、耗时、响应头与响应体。
- [ ] 对错误 URL / 网络失败有友好提示，应用不崩溃。
- [ ] 请求 / 响应主面板的布局与风格基本对标 Postman（请求行 + 配置 Tabs + 响应区）。
- [ ] 提供可复现的开发与构建说明。

## 三、执行计划

### 3.1 方案设计

**技术选型**
- 桌面框架：**Tauri 2.x**（Rust 后端 + 系统 WebView 前端，包体小、安全）。
- 后端：**Rust**；HTTP 客户端 `reqwest`（`rustls` TLS）；序列化 `serde` / `serde_json` + YAML（`serde_norway` 或 `yaml-rust2`）；异步运行时 `tokio`。
- 前端：**React 18 + TypeScript + Vite**（已确定）。
  - 理由：后续 DAG 编排画布可直接复用成熟的 `@xyflow/react`（React Flow）。
  - UI 参考 Postman 紧凑风格，可用组件库（shadcn/ui）或自绘实现。
- 前后端通信：Tauri command + `invoke`。

**核心数据模型**（为 DAG 预留，MVP 仅用到单请求部分）
- `ApiRequest { method, url, headers, body }`
- `ApiResponse { status, status_text, headers, body, elapsed_ms }`
- `CaseNode { id, request }`、`CaseEdge { from, to }`、`Case { nodes, edges }`；单 API case = 1 node / 0 edge，磁盘以 `.yml` 存储。

**目录结构（`app/`）**
```
app/
├── src/                 # 前端（React + TS）
│   ├── App.tsx          # 请求编辑 + 响应展示
│   └── main.tsx
├── src-tauri/           # Rust 后端
│   ├── src/
│   │   ├── main.rs
│   │   └── commands.rs  # send_request 命令
│   ├── Cargo.toml
│   └── tauri.conf.json
├── index.html
├── package.json
└── vite.config.ts
```

**关键交互流程**
前端表单 → `invoke("send_request", { request })` → Rust `send_request()` 用 `reqwest` 发起 → 组装 `ApiResponse` 返回 → 前端渲染。

### 3.2 任务拆解与里程碑

- **M1 脚手架**：在 `app/` 用 `npm create tauri-app@latest`（React + TS 模板）初始化，跑通 `npm run tauri dev` 打开窗口。
- **M2 后端命令**：在 `src-tauri` 实现 `send_request`（`reqwest`），输入 `ApiRequest` 返回 `ApiResponse`，注册到 invoke handler。
- **M3 前端 UI**：请求编辑表单（method/url/headers/body）+ 发送按钮 + 响应展示区。
- **M4 联调闭环**：打通 `前端 → Rust → 真实请求 → 展示`，完善 loading 与错误态。
- **M5 收尾**：编写开发 / 构建说明；将本需求要点合并进 `docs/0.latest/`。

### 3.3 风险与依赖

- **环境依赖**：Rust 工具链、Node.js；Tauri 系统依赖（macOS WebKit / Windows WebView2 / Linux WebKitGTK）。
- **TLS 方案**：`reqwest` 选 `rustls`，避免本地 OpenSSL 依赖，利于跨平台。
- **CORS**：桌面端由 Rust 发请求天然绕过浏览器 CORS；需确保不误用前端 `fetch`。
- **大 / 二进制响应**：MVP 先按文本处理，超大响应与流式留后续优化。
- **YAML 库**：Rust 的 `serde_yaml` 已停维护，采用维护中的 fork（`serde_norway` / `yaml-rust2`）并锁定版本。
- **跨平台打包签名**：留待发布相关需求处理。
